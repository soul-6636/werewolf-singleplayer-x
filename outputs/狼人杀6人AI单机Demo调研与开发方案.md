# 狼人杀 6 人 AI 单机 Demo：规则调研与开发方案

> 目标：先跑通 1 名人类玩家 + 5 名 AI 的单机对局。本文把规则、AI 行为和游戏引擎拆开，便于后续换板子、换模型和加入离线模型。

## 1. 先定口径

“6 人单边局”不是一个所有平台都统一使用的规则名称。狼人杀社区里的“单边”常指只有一名玩家跳预言家（单边预），也有人把它泛指只有好人和狼人两阵营的局。

本方案按以下可执行口径推进：

- 6 名座位：1 名人类玩家 + 5 名 AI；也支持 6 名 AI 自动跑模拟。
- 角色：`2 狼人 + 2 平民 + 1 预言家 + 1 女巫`。
- 阵营：好人、狼人两阵营，没有第三方。
- 初版不设警长、猎人、守卫、白痴等额外角色。
- 初版文本交互，不做语音；桌面壳和本地模型放到闭环跑通之后。

这套角色配置是常见的 6 人入门板子，但不是唯一“官方标准”。公开资料中也能看到 `1 预言家 + 1 猎人 + 2 平民 + 2 狼人`、纯生推等变体；女巫规则、死亡是否翻牌、胜负条件也常由平台自定。

## 2. 已确认事实、设计选择与未知项

### 已确认的通用规则

- 狼人杀/狼人（Werewolf）是隐藏身份的社交推理游戏，核心是好人多数阵营对抗狼人少数阵营。
- 游戏按夜晚和白天交替：夜晚执行秘密技能，白天讨论并投票放逐一名玩家。
- 预言家通常每晚查验一名玩家的阵营，而不是直接得知具体身份。
- 狼人夜间选择一名目标淘汰；好人通常通过淘汰全部狼人获胜，狼人通常在人数达到好人阵营平票或规则指定条件时获胜。

### 本 Demo 的明确设计选择

以下不是唯一正确答案，而是为了让 6 人局更容易测试、复盘和调平衡：

| 项目 | Demo 默认值 | 备注 |
|---|---|---|
| 胜负 | `屠边`：平民全部死亡或神职全部死亡时狼人胜利；狼人全部死亡时好人胜利 | 狼人达到人数平不直接结束 |
| 同时满足 | 狼人全部死亡的好人胜利条件优先 | 作为当前 Demo 的明确裁定，可配置化 |
| 身份信息 | 死亡不翻身份，结算时全揭；开发模式可实时揭示 | 暗牌更有推理价值，开发时需要可观测性 |
| 女巫解药 | 1 瓶，看到狼刀后决定是否救；默认首夜可自救 | “能否自救”是平台差异，必须配置化 |
| 女巫毒药 | 1 瓶，每局一次；默认不能和解药同夜使用 | 目标为任一存活玩家，不允许毒自己 |
| 预言家 | 每晚查验 1 人，返回“好人/狼人” | 不返回“平民/神职”等具体身份 |
| 狼人行动 | 每晚合议 1 个目标 | 2 名 AI 狼人由引擎协调，不共享其他角色的私密上下文 |
| 白天发言 | 存活玩家按座位顺序各发言 1 次 | 首轮发言后直接投票，后续再考虑多轮发言 |
| 投票平票 | 第一次平票只在并列者中重投；再次平票无人出局 | 避免随机淘汰影响调试 |
| 遗言 | 初版关闭 | 遗言会显著改变信息量，后续做配置项 |
| 警长 | 无 | 6 人局不增加额外竞选流程 |

### 暂时无法确认的社区差异

- 不同平台对双方胜利条件同轮触发时的优先级不同；本 Demo 暂定好人优先。
- 女巫能否首夜自救、能否同夜同时使用两瓶药、解药是否能救自己，不存在所有平台一致的答案。
- 死亡是否翻牌、猎人被毒后能否开枪、平票是否流局，均需要以目标平台或自定义规则为准。

## 3. MVP 规则书

### 角色能力

**狼人（2 名）**

- 夜间共同选择一名存活玩家作为刀口。
- 狼人知道另一名狼人是谁。
- 白天伪装成好人发言和投票。

**预言家（1 名）**

- 每晚查验一名其他存活玩家。
- 结果仅为“狼人”或“好人”。女巫、平民和预言家都属于好人阵营。

**女巫（1 名）**

- 有一瓶解药：救活当晚被狼人选中的目标，一局一次。
- 有一瓶毒药：毒杀一名存活玩家，一局一次。
- Demo 默认解药和毒药不能在同一晚使用；是否允许首夜自救由配置决定。

**平民（2 名）**

- 没有夜间技能。
- 通过白天发言、投票和公开事件推理。

### 每日流程

1. **发牌与私密信息**：随机分配角色；狼人互认；预言家和女巫只知道自己的身份。
2. **夜晚**：狼人选刀；女巫看到刀口后决定解药/毒药；预言家查验。UI 可以按固定顺序演出，但引擎在同一晚统一结算。
3. **天亮**：公布昨夜是否有人死亡，不公布杀手和技能细节。
4. **白天发言**：存活玩家按顺序发言；AI 每次只看到自己合法可知的信息。
5. **投票**：每名存活玩家投一名其他存活玩家或弃票；弃票不计入候选票数，再按平票规则处理放逐。
6. **胜负检查**：夜晚结算后、放逐结算后都检查一次，避免多进行一个不必要的阶段。
7. 未结束则进入下一夜。

### 需要写成纯函数的结算规则

```text
resolveNight(wolfTarget, witchSave, witchPoison, guardNone, alivePlayers)
  -> deaths, publicEvents, privateEvents

resolveVote(votes, alivePlayers, tiePolicy)
  -> eliminatedPlayer | null, publicEvents

checkWin(alivePlayers, winPolicy)
  -> NONE | VILLAGERS | WEREWOLVES
```

关键原则：模型不能直接修改 `alivePlayers`、角色或药水数量；模型只能提交意图，规则引擎验证并结算。

当前 `checkWin` 的精确定义：

1. 存活狼人为 0：好人胜利。
2. 否则，存活平民为 0：狼人胜利。
3. 否则，存活神职（预言家、女巫）为 0：狼人胜利。
4. 其他情况继续游戏；狼人和好人人数相等不直接结束。

## 4. AI 对接方案

### 对“单机 + AI”的边界说明

如果接云端大模型，这是一款“本地单人运行”的游戏，不是严格意义上的离线游戏。正式 Web 产品不应在浏览器前端持久化 API Key。当前本机 Demo 按测试需求将 Key 明文保存在浏览器 `localStorage`，并通过本机代理调用模型；该方案仅适合个人电脑测试，设置页必须支持清除。正式版本建议：

```text
React/Vite UI
      |
      v
TypeScript Game Engine  <-->  Event Log / Replay
      |
      v
AI Orchestrator  -->  Provider Adapter  -->  OpenAI-compatible API
                                      \->  Anthropic-compatible API
                                      \->  本地模型（后续）
```

线上 Provider 必须兼容两类 API 方言：OpenAI Chat Completions 兼容格式和 Anthropic Messages 兼容格式。`baseUrl`、路径、模型名、认证头和协议版本全部配置化；上层只使用统一的 `AIProvider` 接口。后续可增加 Ollama 或其他本地推理服务适配器，保持接口不变。

### 每个 AI 都必须有独立视角

每次调用模型时，只注入四类信息：

1. 当前玩家自己的身份和可用技能。
2. 该玩家合法知道的私密信息：狼人队友、预言家查验结果、女巫药水状态。
3. 所有玩家都能看到的公开事件和发言。
4. 当前阶段、存活座位、可选目标和严格的输出 schema。

不能把整局真值状态放进一个共享 prompt，否则 AI 会获得不该知道的角色信息。

### AI 行为拆分

- `speak`：生成 1 段白天发言，限制长度，例如 80～160 个中文字符。
- `vote`：从引擎给出的合法目标或 `ABSTAIN`（弃票）中选择 1 项。
- `seer_check`：预言家选择查验目标。
- `witch_action`：女巫选择 `save / poison / pass`，毒药还要带目标。
- `wolf_kill`：狼人提交候选刀口；由狼队协调器产生最终目标。

不要让一个自由文本调用同时承担发言、技能和投票。结构化动作便于校验、重试和回放。

建议统一输出：

```json
{
  "action": "vote",
  "targetId": "P3",
  "speech": "我更怀疑 P3，因为他的查验逻辑前后不一致。"
}
```

引擎只采信 `action`；`speech` 仅作为展示文本，不能改变规则。

### 提示词原则

- 说明阵营目标和当前阶段，不要求模型输出长篇隐藏推理。
- 只要求短的 `reasoningSummary` 或不输出理由，避免把内部思考当成游戏事实。
- 给出可选目标白名单，禁止模型自行编造座位 ID。
- 解析失败、目标已死亡、技能已用完时，先自动重试一次；仍失败则使用确定性兜底策略并记日志。
- AI 发言中的“我验到某人是狼”只是游戏内声明，不应直接变成引擎事实。

### 性格与难度

初版不要为每个 AI 建复杂长期记忆。使用少量配置即可：

```json
{
  "style": "cautious",
  "verbosity": 0.6,
  "riskTolerance": 0.4,
  "deception": 0.7
}
```

难度首先通过信息处理质量、投票权重和模型参数调节；不要靠给 AI 额外真值信息来“作弊”。

## 5. 推荐技术拆分

### MVP 技术栈

- 前端：React + TypeScript + Vite。
- 游戏核心：独立 TypeScript 包，纯状态机，浏览器和测试都可复用。
- 本地服务：Node.js + Fastify（或同等轻量 HTTP 层），负责模型调用和 SSE 流式发言。
- 校验：Zod 或同等 schema 校验库。
- 存档：首版 JSON/JSONL 事件日志；需要多局统计时再接 SQLite。
- 打包：浏览器闭环稳定后再选 Tauri/Electron。

### 建议目录

```text
src/
  engine/       # 规则、状态机、随机种子、胜负结算
  ai/           # context 构建、provider、输出校验、兜底
  content/      # 角色文案、提示词、AI 性格
  ui/            # 桌面、发言区、行动面板、日志
  replay/        # 事件回放和调试揭示
server/
  routes/       # games、events、ai
  providers/    # cloud、local
tests/
  engine/       # 规则单测和属性测试
  simulations/  # 固定种子对局
```

### 核心数据模型

```ts
type Role = "WEREWOLF" | "VILLAGER" | "SEER" | "WITCH";
type Faction = "VILLAGE" | "WEREWOLF";

type PlayerState = {
  id: string;
  displayName: string;
  controller: "HUMAN" | "AI";
  role: Role;
  alive: boolean;
  witchSaveUsed?: boolean;
  witchPoisonUsed?: boolean;
};

type GameState = {
  seed: number;
  day: number;
  phase: "SETUP" | "NIGHT" | "DAWN" | "DISCUSSION" | "VOTE" | "ENDED";
  players: PlayerState[];
  events: PublicEvent[];
  privateEvents: Record<string, PrivateEvent[]>;
};
```

## 6. 初期 Demo 的最小可验收闭环

### P0：规则引擎

- 固定种子可复现发牌、夜刀、投票和胜负。
- 6 个座位能完整走完至少一局。
- 女巫救人、毒人、药水消耗和冲突顺序有单测。
- 平票重投、无人出局和胜负提前结束有单测。

### P1：单机 UI

- 显示玩家座位、存活状态、当前阶段和公开事件。
- 人类玩家能发言、选技能目标、投票。
- AI 发言逐条出现；AI 思考期间 UI 不阻塞。
- 开发开关可显示全部角色、私密事件和经过截断的 `reasoningSummary`；不展示或索取模型隐藏思维链。

### P2：AI 闭环

- 5 个 AI 按独立视角行动；不共享隐藏真值。
- 所有行动经 schema 校验；非法行动可重试或确定性兜底。
- 支持一次完整对局的事件回放。
- 可切换云端 Provider；当前 Demo 的 API Key 存在浏览器本地存储，生产版迁移到本地服务进程或系统凭据存储。
- 云端 Provider 至少通过 OpenAI-compatible 和 Anthropic-compatible 两套 mock contract test。

### P3：可玩性验证

- 固定种子批量跑 100 局，确认没有死循环、非法目标或状态回滚。
- 记录好人/狼人胜率、平均天数、AI 非法动作率、平均每局模型调用数和延迟。
- 用真实玩家试玩至少 10 局，单独记录“是否理解当前信息”“是否觉得 AI 发言像在玩游戏”两项反馈。

这些数字是验收目标，不是已经验证的事实；胜率是否合理必须基于你最终选定的胜负和女巫规则重新测。

## 7. 后续迭代顺序

1. **先稳定规则**：把女巫规则、翻牌、胜负条件做成 `RuleSet` 配置，禁止散落在 UI 或 prompt 中。
2. **再提高 AI**：加入发言记忆摘要、角色风格、狼队协商、反事实复盘。
3. **再做内容**：多板子、猎人/守卫/警长、剧本和成就。
4. **最后做体验**：语音合成、语音输入、桌面壳、设置页和本地模型。

不建议一开始做语音、多人联机、复杂长期记忆或十几个角色；它们会把问题从“规则和 AI 能否稳定跑通”扩大成多个独立产品问题。

## 8. 风险与对应措施

| 风险 | 处理 |
|---|---|
| 屠边规则导致 6 人对局过短或偏狼 | 保持胜负定义不变，通过模拟数据调整女巫规则、AI 策略和发言轮数 |
| 模型输出非法目标 | 目标白名单 + schema 校验 + 重试 + 确定性兜底 |
| AI 互相泄露私密信息 | 每个玩家单独构建 context；不共享真值状态；事件分 public/private |
| 云端延迟和费用 | 文本长度上限、事件摘要、限制每阶段调用次数；后续增加本地 Provider |
| 兼容 API 能力不一致 | Provider 做能力声明和响应归一化；JSON Schema 不可用时按降级链校验和重试 |
| 玩家误以为 AI 的自称是真实信息 | UI 区分“玩家发言”和“引擎已确认事件”；预言家查验结果只进其私密面板 |
| 规则变更导致回归 | 所有胜负和夜间结算使用纯函数 + 固定种子测试 + 回放快照 |

## 结论

第一版应锁定为：`2 狼 + 2 民 + 预言家 + 女巫`、屠边、无警长、文本 UI、1 人类 + 5 AI、规则引擎权威、AI 只提交结构化意图。当前仍需确认的规则开关是：

1. 女巫首夜能否自救；
2. 死亡是否翻牌；
3. 同轮同时屠边且狼人全部死亡时，是否继续采用好人优先。

这三个选择会显著改变 6 人局的长度、信息量和 AI 难度。
